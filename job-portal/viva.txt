=============================================================
NAUKRIFY JOB PORTAL - VIVA / INTERVIEW PREPARATION GUIDE
=============================================================

This document contains a comprehensive list of questions an interviewer 
might ask you based on your Naukrify Job Portal project. 

-------------------------------------------------------------
1. ARCHITECTURE & TECH STACK
-------------------------------------------------------------
Q1: Can you explain the tech stack used in your project and why you chose it?
A: 
- Frontend: React.js (via Vite) for fast rendering and component-based architecture.
- Styling: Tailwind CSS & Shadcn UI (Radix UI) for rapid, accessible, and customizable styling.
- Authentication: Clerk Auth for robust, out-of-the-box secure authentication with role-based metadata.
- Backend/Database: Supabase (PostgreSQL) for a scalable database, raw SQL support, and built-in Storage buckets.

Q2: Why did you use Vite instead of Create React App (CRA)?
A: Vite uses native ES modules during development, making the local dev server incredibly fast. It also uses Rollup for production builds, which is much faster and produces more optimized bundles than Webpack (which CRA uses).

Q3: How is the flow of data handled from the frontend to the database?
A: The frontend makes asynchronous API calls using custom hooks (like `useFetch`) directly to Supabase REST APIs using the Supabase Javascript Client. Authentication tokens are passed from Clerk to Supabase to verify the user.

-------------------------------------------------------------
2. AUTHENTICATION (CLERK + SUPABASE)
-------------------------------------------------------------
Q4: Why use Clerk specifically, instead of Supabase Auth?
A: Clerk provides an excellent pre-built UI, handles session management seamlessly, social logins, and makes role-based access control (RBAC) via `unsafeMetadata` (like 'candidate' vs 'recruiter') very easy to implement without writing custom backend logic.

Q5: Since you used Clerk for Auth and Supabase for DB, how do they talk to each other securely?
A: I configured a Supabase JWT template inside Clerk. When a user logs in via Clerk, Clerk generates a JWT. My custom Supabase client interceptor (in `supabase.js`) requests this token using `session.getToken({ template: 'supabase' })` and attaches it to the Authorization header. Supabase verifies this JWT to grant access to the database.

Q6: How do you handle Role-Based Access Control (Candidate vs Recruiter)?
A: During the onboarding route (`/onboarding`), the user selects a role. This role is saved directly into the user's Clerk profile as `unsafeMetadata`. In the React components, I access this metadata via Clerk's `useUser()` hook and conditionally render views (like showing the "Post Job" button only to recruiters).

-------------------------------------------------------------
3. DATABASE & SUPABASE CONCEPTS
-------------------------------------------------------------
Q7: What tables do you have in your PostgreSQL database?
A: 
1. `companies`: stores company details and logos.
2. `jobs`: stores job descriptions, requirements, hiring status, and is linked to companies via a foreign key.
3. `saved_jobs`: a junction/mapping table tracking which user saved which job.
4. `applications`: tracks resumes, status (applied, interviewing, etc.), and links candidates to jobs.

Q8: What is RLS (Row Level Security) in Supabase? 
A: Row Level Security is a PostgreSQL feature that restricts which rows in a table a user can SELECT, INSERT, UPDATE, or DELETE. (Note: Mention that during local dev, you temporarily disabled it to fix 403 errors, but in production, you write policies that check the `auth.uid()` against the user's ID to ensure they only see their own data).

Q9: How do you handle file uploads, like Resumes and Company Logos?
A: I used Supabase Storage. I created two public buckets: 'resumes' and 'company-logo'. The frontend converts the uploaded file, sends it to the Supabase storage bucket, gets the public URL back, and stores that URL in the respective PostgreSQL table text column.

-------------------------------------------------------------
4. REACT SPECIFICS (HOOKS, ROUTING)
-------------------------------------------------------------
Q10: I see you created a custom `useFetch` hook. What problem does it solve?
A: It abstracts away the repetitive logic of making async API calls, handling loading states, catching errors, and storing response data. This keeps the UI components much cleaner, as they only need to call `fn()` and read `data` or `loading`.

Q11: How do you protect routes so unauthorized users can't see specific pages?
A: I created a `ProtectedRoute` wrapper component. Inside, it checks if `isLoaded` and `isSignedIn` from Clerk. If the user is not signed in, it redirects them to a sign-in query parameter, which triggers the Clerk SignIn modal. It also checks if the user has completed onboarding (role selection).

Q12: What is the purpose of `useEffect` in your `JobListing` component?
A: `useEffect` is used to trigger the fetching of `getCompanies` and `getJobs` when the user is authenticated. It also re-runs `fnJobs()` whenever the dependencies (location, company_id, or searchQuery strings) change, efficiently re-fetching filtered data.

-------------------------------------------------------------
5. UI & STYLING
-------------------------------------------------------------
Q13: Why Tailwind CSS, and what is Shadcn UI?
A: Tailwind is a utility-first CSS framework that allows styling without leaving the JSX file. Shadcn UI is not a component library you install via npm; it's a collection of accessible component code (layered over Radix UI) that you copy and paste into your project. It gives me full control over the component's internal code while saving me from writing complex accessibility logic from scratch.

6. TRICKY / SCALABILITY QUESTIONS
-------------------------------------------------------------
Q14: What was the biggest challenge in this project and how did you solve it?
A: (Mention the issue with database schema mismatch where the code submitted camelCase `isOpen` but Postgres created lowercased `isopen`! Or mention the Clerk JWT and Supabase integration, as linking a distinct Auth provider with a DB provider requires careful JWT template configuration).

Q15: If this portal hits 100,000 users, what would break first?
A: 
1. Database reads/searches: Searching jobs by string `ILIKE` queries will get slow. I would need to implement Full-Text Search features in Postgres or use an external indexer like Algolia/ElasticSearch.
2. Asset delivery: If millions of resumes or logos are loaded, I would need a CDN in front of the Supabase storage buckets.
3. Pagination: The backend currently fetches jobs as a bulk array. We would need to implement pagination (e.g., LIMIT and OFFSET in Supabase) to prevent huge API payload sizes.

7. CORE JAVASCRIPT BASICS
-------------------------------------------------------------
Q16: What is the difference between `let`, `const`, and `var`?
A: `var` is function-scoped and hoisted (which can lead to weird bugs). `let` and `const` are block-scoped. `const` prevents reassigning the variable after its initial declaration, while `let` allows reassignment.

Q17: What are Promises in JavaScript and why did you use async/await?
A: A Promise is an object representing the eventual completion (or failure) of an asynchronous operation. I used `async/await` (syntactic sugar over Promises) because it makes asynchronous code look and behave a bit more like synchronous code, making it much easier to read and debug than using `.then().catch()` chains.

Q18: What is destructuring in JavaScript? Can you give an example from the project?
A: Destructuring is an ES6 feature that allows extracting values from arrays or properties from objects into distinct variables. For example, in the app: `const { user, isLoaded } = useUser();` extracts the `user` and `isLoaded` properties directly from the object returned by the `useUser` hook.

-------------------------------------------------------------
8. REACT INTERVIEW BASICS
-------------------------------------------------------------
Q19: Explain the Virtual DOM.
A: React creates an in-memory data structure cache (the Virtual DOM). When a component's state changes, React updates the Virtual DOM first, then compares it (diffing algorithm) with the actual DOM. It then selectively updates only the parts of the actual browser DOM that changed, which makes React fast.

Q20: What is the difference between State and Props?
A: `props` (short for properties) are passed from a parent component to a child and are read-only (immutable) in the child. `state` is managed within the component itself and can be updated using a setter function (like `setSearchQuery()`), which triggers a re-render.

Q21: Why do we use `key` in lists in React? (e.g., when mapping jobs)
A: Keys help React identify which items have changed, are added, or are removed. Without a unique `key` (like `job.id`), React will re-render the entire list instead of just tweaking the specific modified item, which hurts performance and can cause UI bugs.

Q22: Mention the React hooks you used and their purpose.
A: 
- `useState`: To manage local component state (e.g., search queries, hiring status).
- `useEffect`: To perform side effects (e.g., fetching data when the component mounts or when a dependency changes).
- Custom Hooks (`useFetch`): To encapsulate and reuse loading, data, and error logic across multiple components.

-------------------------------------------------------------
9. CSS & TAILWIND BASICS
-------------------------------------------------------------
Q23: What is the CSS Box Model?
A: Every element on a webpage is a rectangular box. The box model consists of four areas: Content (the actual text/image), Padding (space inside the border), Border (the edge), and Margin (space outside the border).

Q24: What is the difference between Flexbox and CSS Grid? 
A: Flexbox is primarily for one-dimensional layouts (either a row or a column), dealing with alignment and spacing along a single axis. CSS Grid is for two-dimensional layouts, handling both rows and columns simultaneously.

Q25: Why do you prefer Tailwind over standard CSS/SCSS?
A: Tailwind reduces context switching (I don't have to bounce between .css and .jsx files) and ensures consistency. It avoids issues like global CSS conflicts or writing repetitive code by relying on utility classes, which leads to smaller production CSS bundles because it aggressively purges unused styles.
